# Методика НИР-2 TeleBid

## Тема

Исследование архитектурных решений обработки конкурентных торгов и синхронизации состояния в веб-платформах прямых и обратных аукционов.

## Результат НИР-2

Результатом этапа является не готовый научный вывод о «лучшем» транспорте, а воспроизводимый экспериментальный стенд, методика измерений и начальный датасет. Расширенная серия прогонов, статистическая проверка различий и рекомендации веб-разработчикам относятся к НИР-3.

## Научная проблема

В аукционном веб-приложении несколько пользователей одновременно меняют одно состояние. В это же время браузерные клиенты получают обновления с разной задержкой, теряют соединение и повторяют запросы. Из этого следуют пять проверяемых проблем:

1. Сервер может определить неправильного лидера при конкурентной записи ставок.
2. Polling, SSE и WebSocket дают разную задержку и служебную нагрузку.
3. После разрыва клиент может остаться на устаревшей версии или пропустить события.
4. Повтор команды или уведомления может второй раз изменить либо неверно представить состояние.
5. При сворачивании или закрытии Telegram Mini App постоянное соединение отсутствует, поэтому live-событие не гарантирует получение участником результата торгов.

Прикладным сценарием служит Telegram Mini App для прямых аукционов рекламных слотов и обратных аукционов рекламных кампаний. Telegram не является предметом сравнения: он задаёт реалистичный короткий жизненный цикл веб-клиента и внешний канал уведомлений.

## Объект и предмет

Объект исследования: веб-приложения реального времени с конкурентным взаимодействием пользователей.

Предмет исследования: архитектурные методы атомарной обработки команд, доставки версионированных событий, восстановления состояния браузерного клиента после reconnect и надёжной двухканальной доставки уведомлений.

## Цель

Спроектировать и пилотно апробировать стенд для воспроизводимого сравнения polling, SSE и WebSocket, а также проверки корректности конкурентных торгов, идемпотентности команд, версионированного восстановления состояния и доставки уведомлений при закрытом Mini App.

## Гипотезы пилотной серии

- H1. Транзакционная обработка ставки и идемпотентный ключ `(auctionId, commandId)` сохраняют правильные цену и победителя при конкурентных запросах и не допускают повторного эффекта команды.
- H2. В непрерывном соединении SSE и WebSocket дают меньшую p95-задержку доставки события, чем периодический HTTP polling.
- H3. Монотонные версии и replay после reconnect приводят все клиенты к текущей серверной версии без итоговых пропусков событий.
- H4. Hybrid recovery возвращает то же итоговое состояние, что snapshot и replay, выбирая меньший JSON payload.
- H5. Транзакционно сохраняемый inbox Mini App и Telegram Bot fallback обеспечивают получение уведомления после reconnect и временного отказа Bot API без потери, повторного показа и нарушения причинного порядка.

Статус `SUPPORTED` в отчёте означает поддержку в пилотных наблюдениях, а не доказанную статистическую значимость.

## Реализация стенда

```text
Генератор ставок ───────────────► NestJS API ─────────► PostgreSQL
                                      │                     │
                                      │ HTTP / SSE / WS     ├─ журнал событий
                                      ▼                     ├─ команды
Виртуальные браузеры ◄────────── Toxiproxy                  └─ durable inbox
  polling / SSE / WebSocket       latency, jitter, disconnect       │
         Mini App observer ◄──── cursor replay                       └──► Bot dispatcher ──► Telegram
```

Торговые команды идут в контрольный API напрямую. Наблюдающие клиенты и модель Telegram Mini App подключаются к тому же API через Toxiproxy. Поэтому задержка и разрыв влияют на синхронизацию клиентов, но не подменяют результат конкурентной обработки ставок сетевой ошибкой генератора. Запись доменного события и уведомлений выполняется в одной транзакции; доставка ботом происходит после commit и может безопасно повторяться.

Стенд запускается отдельно от Telegram-продукта:

- отдельный Compose project `telebid-research`;
- отдельная PostgreSQL-база и volume;
- API на `localhost:8081`;
- Toxiproxy API на `localhost:8475`;
- исследуемый сетевой канал на `localhost:8667`.

## Сравниваемые способы синхронизации

### Polling

Клиент через заданный интервал выполняет `GET /sync`, передавая последнюю применённую версию. Время ожидания следующего запроса входит в задержку доставки события.

### Server-Sent Events

Клиент держит однонаправленный HTTP-поток. После reconnect сервер учитывает `Last-Event-ID` и отдаёт события после последней версии.

### WebSocket

Клиент получает live-события через Socket.IO. При каждом подключении он отправляет `lastAppliedVersion`, после чего сервер выполняет replay и продолжает поток событий.

## Управляемые факторы

- вид аукциона: прямой или обратный;
- количество браузерных клиентов на каждый транспорт;
- количество ставок и параллельность запросов;
- интервал между волнами команд;
- интервал polling;
- задержка и jitter сети;
- момент и длительность разрыва;
- доля повторно отправленных принятых команд;
- число повторов сценария;
- seed генерации идентификаторов команд.

## Измеряемые показатели

### Корректность торгов

- совпадение фактических цены и победителя с лучшим отправленным предложением;
- непрерывность последовательности версий;
- число повторных эффектов одного `commandId`;
- доля повторов, распознанных как идемпотентный replay.

### Синхронизация клиентов

- p50, p95 и максимум задержки события;
- число итоговых пропусков и дубликатов доставки;
- число клиентов с замеченным разрывом версий;
- доля клиентов, достигших финальной серверной версии;
- p95 времени восстановления после включения сети;
- JSON payload на клиента;
- число запросов, reconnect и сетевых ошибок.

### Восстановление и уведомления

- равенство итоговой версии snapshot, replay и hybrid;
- непрерывность replay;
- соответствие решения hybrid меньшему payload;
- дубликаты, потерянные причинные связи и cursor уведомлений;
- число live- и replay-уведомлений Mini App, потери и нарушения порядка показа;
- число попыток Telegram-доставки и p95 времени до статуса `DELIVERED`.

## Профили

`quick` выполняет четыре smoke-прогона: два вида аукциона в стабильной сети и после мобильного reconnect.

`pilot` выполняет три сценария для двух видов аукциона с тремя повторами по умолчанию:

1. `baseline-live`: 2 клиента на транспорт, 24 ставки, стабильная сеть без разрыва.
2. `mobile-reconnect`: 5 клиентов на транспорт, 48 ставок, задержка 120 ± 80 мс и разрыв 800 мс.
3. `final-burst-unstable`: 10 клиентов на транспорт, 96 ставок, задержка 300 ± 200 мс и разрыв 1400 мс.

Параметры и число повторов сохраняются в `manifest.json`; повтор можно изменить аргументом `--repeats` или переменной `RESEARCH_REPEATS`.

## Протокол одного прогона

1. Сбросить toxics и установить заданные latency и jitter.
2. Создать новый аукцион в изолированной базе.
3. Подключить одинаковое количество polling-, SSE- и WebSocket-клиентов.
4. Сгенерировать монотонно улучшающиеся предложения с фиксированными `commandId`.
5. Отправить предложения конкурентными волнами.
6. При reconnect-сценарии отключить только проксируемый канал клиентов, продолжая принимать ставки сервером.
7. Повторно отправить часть уже принятых команд и завершить аукцион, пока Mini App отключён.
8. Имитировать временный отказ первой попытки Telegram Bot API и проверить повторную доставку.
9. Включить сеть и дождаться сходимости состояния и replay уведомлений либо тайм-аута.
10. Сопоставить клиентские журналы с авторитетным журналом PostgreSQL и проверить snapshot, replay, hybrid, cursor и причинный порядок.

## Обработка данных

Для каждой серии рассчитываются количество наблюдений, минимум, максимум, среднее, медиана, p95, выборочное стандартное отклонение и bootstrap 95% доверительный интервал среднего. Bootstrap использует 2000 детерминированных повторных выборок.

В НИР-2 эти показатели описывают начальный датасет и проверяют пригодность стенда. Для НИР-3 необходимо заранее определить требуемое число повторов, проверить распределения, оценить величину эффекта и применить подходящий критерий для связанных серий измерений.

## Артефакты

Каждый запуск создаёт каталог `datasets/<run-id>/`:

- `events.csv` — сырые доставки событий каждому клиенту;
- `commands.csv` — исходные и повторные попытки ставок;
- `notifications.csv` — получение и безопасный показ каждого уведомления моделью Mini App;
- `clients.csv` — показатели каждого виртуального браузера;
- `trials.csv` — итог каждого транспорта в каждом прогоне;
- `aggregates.csv` — описательная статистика серий;
- `summary.json` — полные структурированные результаты;
- `manifest.json` — среда, commit, конфигурация и схема датасета;
- `report.md` — автоматически сформированный отчёт пилотной проверки.

Описание столбцов находится в `docs/dataset-schema.md`.

## Роль экранной модели

Экран `/?mode=lab` и API `/experiments/run` воспроизводят базовую и предлагаемую архитектуры в детерминированной модели. Они нужны для демонстрации причин lost update, рассинхронизации и повторной обработки команды. Результаты этой модели не смешиваются с основным сетевым датасетом и не заменяют измерения реального API.

## Ограничения пилота

- виртуальные браузеры реализованы Node-клиентами тех же сетевых протоколов, а не отдельными процессами Chromium;
- измеряется JSON payload без HTTP/TLS-заголовков и служебных WebSocket-фреймов;
- Toxiproxy изменяет TCP-канал, но не моделирует все особенности мобильного радио и фонового режима ОС;
- исследовательский контур заменяет внешний Telegram Bot API детерминированным адаптером с тем же интерфейсом и искусственным первым отказом; реальная доставка зависит также от доступности Telegram;
- один локальный компьютер одновременно размещает API, базу и генератор нагрузки;
- пилотная серия недостаточна для универсальных рекомендаций без расширения выборки в НИР-3.
