# Дизайн исследования TeleBid

## Контекст

TeleBid используется как Telegram Mini App. Пользователь открывает торги из канала или чата с ботом, делает ставку и может сразу свернуть приложение. В отличие от обычной постоянно открытой веб-страницы, здесь жизненный цикл интерфейса короткий, а WebSocket-соединение не является надёжным каналом доставки результата.

Прикладной сценарий содержит прямой аукцион рекламного слота и обратный аукцион рекламной кампании. В обоих случаях итог зависит от порядка обработки конкурентных предложений.

## Научная проблема

Нужно обеспечить три свойства одной веб-системы:

1. Корректность торгов при одновременных ставках и повторной отправке запросов.
2. Согласованность состояния Mini App после потери связи и повторного открытия.
3. Доставку значимого события участнику, даже если Mini App закрыт.

Один WebSocket решает только быструю доставку во время активного соединения. Он не сериализует изменения в базе, не восстанавливает пропущенные события сам по себе и не уведомляет закрытый клиент.

## Объект и предмет

Объект исследования: веб-приложения реального времени с конкурентным взаимодействием пользователей на платформе Telegram.

Предмет исследования: архитектурные методы атомарной обработки команд, синхронизации клиентского состояния после reconnect и гибридной доставки событий через Telegram Mini App и Bot API.

## Сравниваемые архитектуры

### A. Базовая

1. Каждый запрос отдельно читает текущую цену.
2. Допустимая ставка записывается без последовательной обработки конкурирующих команд.
3. Новое состояние отправляется только подключённым WebSocket-клиентам.
4. Повтор запроса может повторно повлиять на торги.
5. После reconnect клиент не запрашивает пропущенные версии.

Такая реализация проста, но допускает lost update, устаревший экран и потерю уведомления при закрытом Mini App.

### B. Предлагаемая

1. Ставка обрабатывается в транзакции с блокировкой состояния конкретного аукциона.
2. `commandId` обеспечивает идемпотентность повторного запроса.
3. Каждое принятое изменение получает монотонную версию.
4. Доменное событие сохраняется в журнале.
5. WebSocket доставляет новые события активному Mini App.
6. После reconnect клиент передаёт последнюю применённую версию и получает replay либо snapshot.
7. Уведомление сохраняется независимо от соединения и отправляется через Telegram Bot API с повторными попытками.
8. Уведомление связано с версией состояния и не должно отображаться раньше соответствующего изменения торгов.

## Гипотеза

Применение атомарной обработки ставок, идемпотентных команд, версионированного восстановления Mini App и сохраняемой доставки уведомлений через Telegram-бота статистически уменьшает долю неверных результатов торгов, устаревших клиентских состояний, потерянных уведомлений и повторно применённых команд по сравнению с параллельной обработкой и доставкой только через активное WebSocket-соединение.

Компромисс предлагаемого решения состоит в возможном увеличении p95 задержки команды из-за ожидания транзакционной обработки при финальном всплеске ставок.

## Экспериментальная модель

Один сгенерированный поток команд с фиксированным seed подаётся в обе реализации. Для каждого сценария заранее известны лучшая допустимая цена и правильный победитель. Прогоны повторяются с разными последовательностями сетевых задержек.

Управляемые факторы:

- прямой или обратный аукцион;
- количество участников;
- количество ставок;
- ширина временного окна всплеска;
- базовая задержка и jitter сети;
- доля разрывов соединения;
- доля повторно отправленных команд;
- количество повторных прогонов;
- seed генератора.

## Метрики

- доля прогонов с неправильным победителем;
- доля клиентов с неактуальным состоянием после торгов;
- доля недоставленных значимых уведомлений;
- доля повторных команд, повторно изменивших состояние;
- p95 времени обработки команды;
- пропускная способность;
- среднее количество принятых предложений.

## Критерий пилотной проверки

Гипотеза считается поддержанной в пилотном эксперименте, если предлагаемая архитектура улучшает не менее трёх из четырёх показателей корректности, а во всех базовых сценариях не допускает неверного победителя, рассинхрона, потери уведомления и повторного эффекта команды.

НИР-2 создаёт стенд, методику и начальный датасет. Статистическое сравнение серий экспериментов, оценка значимости различий и рекомендации веб-разработчикам относятся к следующему этапу исследования.

## Технологическая реализация стенда

- React и TypeScript для Telegram Mini App;
- Telegram Web Apps API для идентификации пользователя и нативного интерфейса;
- NestJS для HTTP API, WebSocket и фоновых обработчиков;
- PostgreSQL и TypeORM для транзакций, версий, команд, событий и уведомлений;
- Telegram Bot API для публикации карточек и внешней доставки событий;
- Docker Compose для воспроизводимого запуска;
- TypeScript load runner для генерации датасета JSON, CSV и Markdown.
