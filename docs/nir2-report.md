# ВВЕДЕНИЕ

Веб-приложения реального времени часто предоставляют нескольким пользователям доступ к одному изменяемому объекту. В аукционной системе таким объектом является состояние торгов: текущая цена, лидер, срок окончания и итоговый победитель. Каждая ставка должна быть проверена относительно состояния, актуального в момент её обработки, а результат должен быстро и однозначно отобразиться на всех подключённых клиентах. Если две команды приходят почти одновременно, простое чтение цены и последующая запись нового значения могут привести к lost update, неверному лидеру или нарушению порядка событий.

Сетевой транспорт не решает эту проблему сам по себе. WebSocket предоставляет постоянный двунаправленный канал [1], SSE обеспечивает серверный поток поверх HTTP и автоматическое переподключение EventSource [2], а polling периодически запрашивает изменения. Однако при разрыве соединения любой клиент может пропустить подтверждённые сервером события. Документация Socket.IO прямо разделяет порядок сообщений и гарантию их прибытия: по умолчанию сервер не хранит сообщения для отключённого клиента, поэтому усиление доставки требует уникальных идентификаторов, журнала событий и клиентского offset [3]. Даже встроенное восстановление Socket.IO не отменяет необходимость прикладной синхронизации, если восстановление сессии невозможно [4].

Для Telegram Mini App проблема проявляется особенно наглядно. Mini App является веб-интерфейсом внутри Telegram и получает подтверждённые сервером данные пользователя через `initData`, которое необходимо проверять на сервере [5]. Пользователь может в любой момент свернуть или закрыть интерфейс. В этот момент WebSocket перестаёт быть надёжным способом сообщить, что ставка перебита или аукцион завершён. Поэтому рабочая архитектура должна отделять живое обновление экрана от долговременного уведомления: активный клиент получает событие через WebSocket, отключённый клиент восстанавливает inbox по cursor, а Telegram-бот независимо доставляет критичное сообщение через Bot API [6].

Тема работы: «Исследование архитектурных решений обработки конкурентных торгов и синхронизации состояния в веб-платформах прямых и обратных аукционов». Аукцион используется не как объект экономического исследования, а как воспроизводимая модель событийного веб-приложения со строгими инвариантами, высокой концентрацией команд и кратковременными отключениями клиентов. Прямой аукцион повышает цену рекламного слота, обратный аукцион снижает стоимость выполнения рекламной кампании; при этом транспорт, протокол событий и механизм восстановления остаются общими.

Актуальность работы определяется тремя обстоятельствами. Во-первых, популярная формулировка «использовать WebSocket для real-time» не отвечает на вопросы о пропущенных событиях, повторных командах и согласованности после reconnect. Во-вторых, сравнение транспорта без фиксации серверной семантики смешивает сетевую задержку с ошибками конкурентной записи. В-третьих, Telegram Mini App создаёт практический сценарий, в котором веб-клиент регулярно становится неактивным, а уведомление должно переживать его жизненный цикл.

Научная проблема состоит в недостаточной определённости границ применимости архитектурных решений при одновременном изменении интенсивности команд, количества клиентов, задержки сети и длительности разрыва. Спецификации определяют свойства каналов, исследования сравнивают отдельные транспортные механизмы [10–12], а теория распределённых систем описывает порядок и сходимость [13–17]. Для рассматриваемого класса приложений требуется единый стенд, который разделяет корректность серверной обработки, live-доставку, восстановление состояния и доставку уведомлений.

Объект исследования: веб-платформы реального времени с конкурентным взаимодействием пользователей.

Предмет исследования: архитектурные методы атомарной обработки команд, доставки версионированных событий, восстановления согласованного состояния веб-клиента и двухканальной доставки уведомлений в Telegram Mini App.

Цель НИР-2: спроектировать, реализовать и пилотно апробировать воспроизводимый экспериментальный стенд для исследования конкурентных торгов, способов синхронизации веб-клиентов и восстановления после разрыва соединения.

Для достижения цели решаются следующие задачи:

1. уточнить границы исследования и разделить серверную обработку, транспорт, recovery и уведомления;
2. определить инварианты прямого и обратного аукциона и требования к идемпотентности;
3. реализовать общий журнал версионированных событий и три транспорта доставки;
4. реализовать snapshot, replay и hybrid recovery;
5. разработать durable inbox Mini App и Telegram Bot fallback с повторной доставкой;
6. построить управляемый генератор нагрузки и сетевых нарушений;
7. определить метрики, формат датасета и критерии поддержки гипотез;
8. провести пилотную серию и сформулировать ограничения и план НИР-3.

Методы исследования включают анализ научных и технических источников, структурно-функциональное моделирование, контролируемый вычислительный эксперимент и описательную статистику. Практическим результатом является TypeScript-монорепозиторий TeleBid, отдельный Docker-контур эксперимента, воспроизводимый сценарий запуска и машиночитаемый датасет.

# 1 АНАЛИЗ СОСТОЯНИЯ ПРОБЛЕМЫ

## 1.1 Разделение корректности торгов и синхронизации клиента

В распределённом веб-приложении полезно различать авторитетное состояние и клиентское представление. Авторитетное состояние принадлежит серверу и базе данных. Браузер хранит производную реплику, которая может временно отставать, но после восстановления обязана прийти к той же версии. Такое разделение следует из классических работ о порядке событий и репликации [13–16].

Для аукциона недостаточно обеспечить eventual consistency между браузерами. Ставка является командой с предусловием: для прямого аукциона новая цена должна быть выше текущей не менее чем на шаг, для обратного — ниже. Если клиенты самостоятельно объединяют конкурирующие изменения, они могут принять взаимоисключающие решения. Поэтому CRDT-подходы полезны для обозначения границы исследования, но серверный выбор победителя должен быть сериализован [17]. Клиенты получают уже подтверждённый порядок и воспроизводят его.

В работе выделены четыре слоя архитектуры:

1. Command layer принимает ставку и проверяет её в транзакции.
2. Event layer фиксирует подтверждённое изменение с монотонной версией.
3. Sync layer доставляет изменения и восстанавливает реплику клиента.
4. Notification layer сохраняет пользовательское сообщение и доставляет его через активный Mini App либо Telegram-бота.

Ошибка каждого слоя измеряется отдельно. Неверный победитель является ошибкой обработки команды. Пропущенная версия является ошибкой синхронизации. Устаревший экран после reconnect является ошибкой recovery. Недоставленное сообщение при закрытом Mini App является ошибкой контура уведомлений. Такое разделение исключает ситуацию, в которой быстрый транспорт маскирует некорректную бизнес-логику.

## 1.2 Прямые и обратные аукционы как нагрузочная модель

В прямом аукционе участники повышают цену, а победителем становится максимальное допустимое предложение. В прикладном сценарии TeleBid владелец Telegram-канала выставляет рекламный слот, а рекламодатели конкурируют за размещение. В обратном аукционе рекламодатель публикует задачу, владельцы каналов снижают стоимость её выполнения, и лидирует минимальная допустимая цена.

С точки зрения веб-архитектуры оба механизма имеют одинаковый жизненный цикл. Различается только предикат `isBetter`. Это позволяет повторять одну экспериментальную конфигурацию для двух направлений изменения цены и проверять, не зависит ли корректность реализации от знака сравнения.

Аукционы характеризуются всплесками активности перед завершением. Исследования поведения участников показывают концентрацию ставок в конце торгов [18]. В настоящей работе экономическая причина такого поведения не изучается; финальный всплеск используется как технический профиль нагрузки, создающий конкурентные запросы и одновременно большое число событий для синхронизации.

Инварианты торгов сформулированы следующим образом:

- статус `CLOSED` запрещает новые ставки;
- цена изменяется только в допустимом направлении;
- каждая принятая команда увеличивает `aggregateVersion`;
- в одной версии существует не более одного доменного события;
- итоговый лидер соответствует текущей цене;
- один `(auctionId, commandId)` создаёт не более одного эффекта;
- закрытие создаёт финальное событие и уведомления в той же транзакционной границе.

## 1.3 Polling, SSE и WebSocket

HTTP polling периодически выполняет запрос `GET /auctions/{id}/sync?sinceVersion=v`. Его достоинства — простая модель отказа и естественное повторение запроса. Основной недостаток — окно устаревания, ограниченное интервалом polling, и повторные HTTP-запросы при отсутствии изменений. В эксперименте используется keep-alive окружения Node.js; поэтому polling не трактуется как обязательное создание нового TCP-соединения на каждый запрос.

SSE использует стандартный интерфейс EventSource и формат `text/event-stream`. Спецификация предусматривает автоматическое переподключение и поле `Last-Event-ID` [2]. Сервер TeleBid присваивает этому идентификатору значение `aggregateVersion`, загружает события после cursor и буферизует live-события на границе начального replay. Канал однонаправленный, поэтому команды ставок, как и в остальных вариантах, передаются отдельным HTTP POST.

WebSocket создаёт постоянный двунаправленный канал [1]. В реализации Socket.IO клиент отправляет `auctionId`, `lastAppliedVersion` и выбранную стратегию recovery. Сервер отвечает последовательностью `sync:begin`, `sync:payload`, `sync:complete`, после чего продолжает live-поток. События, пришедшие во время синхронизации, временно буферизуются и применяются после recovery в порядке версий.

Таблица 1 – Сравниваемые транспортные механизмы

| Транспорт | Получение обновлений | Восстановление после разрыва | Основной компромисс |
| --- | --- | --- | --- |
| HTTP polling | Периодический sync-запрос | Следующий запрос с `sinceVersion` | Простота и предсказуемое возобновление ценой окна устаревания |
| SSE | Однонаправленный поток EventSource | `Last-Event-ID` и журнал событий | Низкая live-задержка, но отдельный канал команд и задержка reconnect |
| WebSocket | Постоянный двунаправленный канал | Повторная подписка с последней версией | Низкая live-задержка, но recovery остаётся прикладной обязанностью |

Важный вывод анализа состоит в том, что транспорт не равен синхронизации. Socket.IO по умолчанию гарантирует порядок прибывших сообщений, но не повторяет серверные события, пропущенные отключённым клиентом [3]. Поэтому все три варианта в стенде опираются на общий журнал PostgreSQL и одну версионную модель.

## 1.4 Snapshot, replay и hybrid recovery

Snapshot передаёт клиенту полное актуальное состояние и серверную версию. Он не требует последовательного применения истории, но повторно передаёт поля, которые не изменились. Replay передаёт события из диапазона `(v_client; v_server]`. Он сохраняет историю изменений и обычно выгоден при коротком разрыве, но его размер растёт вместе с числом пропущенных событий.

Hybrid recovery в реализованном прототипе сначала проверяет непрерывность доступного журнала, затем сравнивает размер сериализованного JSON snapshot и replay. Выбирается меньший корректный payload. Это намеренно более простая политика, чем модель полной стоимости с сетевой пропускной способностью и временем клиентского reducer. НИР-2 проверяет работоспособность измерительного контура; временная cost model относится к следующему этапу.

Таблица 2 – Стратегии восстановления

| Стратегия | Передаваемые данные | Условие применимости | Риск |
| --- | --- | --- | --- |
| Snapshot | Полное состояние на `serverVersion` | Всегда доступно при существующем аукционе | Избыточный payload для большого состояния |
| Replay | Непрерывный диапазон событий | Журнал содержит все версии после cursor | Рост payload и стоимости reducer при длинном разрыве |
| Hybrid | Меньший из корректных snapshot/replay | Доступны состояние и журнал | Размер JSON не учитывает CPU и сетевые заголовки |

Клиент хранит последнюю применённую версию и множество `eventId`. Событие с уже известным идентификатором не изменяет состояние повторно. Версия `currentVersion + 1` применяется, большая версия сигнализирует о gap и запускает recovery. Snapshot принимается как новая базовая реплика. После завершения синхронизации буфер live-событий сортируется по версии.

## 1.5 Жизненный цикл Telegram Mini App и уведомления

Telegram Mini App запускается внутри клиента Telegram и получает контекст пользователя. Сервер не должен доверять `initDataUnsafe`: официальная документация требует передавать строку `initData` на backend и проверять подпись [5]. В TeleBid этот контекст используется как идентичность владельца аукциона, участника и получателя уведомлений; demo-аутентификация разрешена только в локальном и исследовательском контуре.

Активность Mini App не гарантирована. Пользователь может открыть каталог, сделать ставку и закрыть интерфейс до окончания торгов. Постоянное соединение разрывается, а сервер не может считать событие доставленным только потому, что вызван `socket.emit`. Поэтому уведомление проектируется как самостоятельная сохраняемая сущность, причинно связанная с доменным событием.

Предлагаемый контур состоит из трёх путей:

1. Live WebSocket доставляет уведомление активному Mini App с минимальной задержкой.
2. Durable inbox хранит `notificationId`, глобальный `sequence`, `eventId` и `aggregateVersion`; после открытия Mini App запрашивает элементы после cursor.
3. Telegram Bot dispatcher отправляет сообщение через `sendMessage`, сохраняет статус и повторяет временно неуспешную попытку с экспоненциальной задержкой.

Минимальная семантика доставки — at-least-once на уровне каналов и exactly-once effect на уровне интерфейса. Повтор одного `notificationId` может прийти по live и replay, но не должен второй раз появиться в ленте. Кроме того, уведомление о версии `v` нельзя показывать как подтверждённый итог до того, как клиент применил состояние не ниже `v`. Поэтому Mini App временно удерживает уведомления, опережающие состояние, и освобождает их после синхронизации.

## 2.1 Исследовательский вопрос и гипотезы

Исследовательский вопрос сформулирован следующим образом: какие архитектурные решения позволяют сохранить корректный результат конкурентных торгов, привести веб-клиенты к актуальному состоянию после разрыва и доставить причинно связанное уведомление при закрытом Telegram Mini App?

Пилотная серия проверяет пять гипотез.

H1. Транзакционная обработка ставки и идемпотентный ключ `(auctionId, commandId)` сохраняют правильные цену и победителя при конкурентных запросах и не допускают повторного эффекта команды.

H2. В непрерывном соединении SSE и WebSocket дают меньшую p95-задержку live-доставки события, чем HTTP polling с фиксированным интервалом.

H3. Монотонные версии и replay после reconnect приводят все клиенты к финальной серверной версии без итоговых пропусков событий.

H4. Hybrid recovery возвращает то же итоговое состояние, что snapshot и replay, и выбирает вариант с меньшим сериализованным JSON payload.

H5. Транзакционно сохраняемый inbox Mini App и Telegram Bot fallback обеспечивают получение уведомления после reconnect и временного отказа Bot API без потери, повторного показа и нарушения причинного порядка.

H1, H3 и H5 преимущественно проверяют инварианты: одно нарушение в тестируемом диапазоне означает, что архитектура не удовлетворяет критерию. H2 и дальнейшее сравнение времени recovery требуют статистического расширения выборки. Статус `SUPPORTED` в НИР-2 означает поддержку гипотезы пилотными наблюдениями, а не универсальное доказательство.

## 2.2 Факторы и контролируемые условия

Эксперимент разделяет контрольный и наблюдаемый каналы. Генератор отправляет торговые команды напрямую в API. Виртуальные веб-клиенты подключаются к тому же API через Toxiproxy. Задержка и разрыв изменяют доставку состояния, но не подменяют результат конкурирующих команд случайными сетевыми ошибками генератора.

## 2.3 Зависимые показатели и критерии

Корректность торгов оценивается совпадением фактической цены и лидера с лучшим сформированным предложением, непрерывностью авторитетного журнала, числом повторных эффектов `commandId` и долей повторов, распознанных как идемпотентный replay.

Для каждого виртуального клиента фиксируются уникальные полученные события, пропуски относительно авторитетного журнала, дубликаты доставки, наблюдавшиеся разрывы версий, число запросов и reconnect, объём JSON payload, p50, p95 и максимум задержки. Сходимость означает равенство финальной клиентской и серверной версии. Время recovery измеряется от включения прокси до достижения целевой версии.

Для уведомлений разделяются три времени: создание записи в транзакции, получение Mini App и безопасный показ после достижения связанной `aggregateVersion`. Дополнительно фиксируются источник `live` или `replay`, дубликат доставки, причинное опережение состояния, число попыток Bot API и время до `DELIVERED`.

| --- | --- |
| H1 | Во всех trial правильны цена и победитель; журнал непрерывен; каждый намеренный retry распознан; повторных эффектов нет |
| H2 | В каждой baseline-группе медиана trial-p95 SSE и WebSocket ниже polling |

## 2.4 Статистическая обработка

Для каждой комбинации сценария, вида торгов и транспорта рассчитываются число наблюдений, минимум, максимум, среднее, медиана, p95, выборочное стандартное отклонение и bootstrap 95-процентный доверительный интервал среднего. Bootstrap использует 2000 повторных выборок с детерминированным seed. В H2 сравнивается медиана trial-p95, а не отдельные сетевые пакеты, чтобы единицей анализа оставался независимый запуск.

Три повтора достаточны для smoke-проверки измерительного контура, но недостаточны для устойчивого вывода о распределении. В НИР-3 число повторов должно определяться по наблюдаемой вариативности и минимально значимому эффекту. Поскольку одинаковые workload повторяются для нескольких транспортов, для основного сравнения планируется критерий Фридмана, парные сравнения критерием Уилкоксона с поправкой Холма и оценка размера эффекта. Для долей ошибок следует использовать интервалы Уилсона и парные критерии для бинарных исходов. Если предпосылки параметрических методов будут выполнены, результаты могут быть дополнительно проверены моделью со смешанными эффектами.

Инварианты корректности не сводятся к p-value. Например, единичный повторный эффект одной финансово значимой команды является архитектурным дефектом даже при низкой доле. Поэтому статистические показатели дополняют, но не заменяют проверку условий безопасности.

## 2.5 Угрозы валидности

Внутренняя валидность поддерживается общим серверным ядром, единым envelope и одинаковым потоком команд. Сетевые нарушения применяются только к наблюдающим клиентам. Seed и конфигурация сохраняются в `manifest.json`.

Основные ограничения следующие:

- виртуальные клиенты реализованы Node.js библиотеками протоколов, а не отдельными экземплярами Chromium;
- JSON payload не включает HTTP-заголовки, TLS и служебные WebSocket-фреймы;
- Toxiproxy моделирует TCP-задержку и разрыв, но не все особенности мобильной ОС;
- API, база, генератор и клиенты работают на одном компьютере;
- Telegram Bot API в исследовательском контуре заменён детерминированным адаптером с тем же вызовом `sendMessage`;
- тестируется один экземпляр API, без межузлового fan-out;
- малое состояние аукциона ограничивает исследование границы replay/snapshot.

Внешняя валидность ограничена событийными веб-приложениями с авторитетным сервером. Абсолютные миллисекунды нельзя без проверки переносить на другую инфраструктуру, однако методика, инварианты и формат данных являются воспроизводимыми.

# 3 РЕАЛИЗАЦИЯ ЭКСПЕРИМЕНТАЛЬНОГО СТЕНДА

## 3.1 Состав программного решения

TeleBid реализован как TypeScript-монорепозиторий. Пользовательская часть представляет React/Vite Telegram Mini App. Серверная часть построена на NestJS и использует PostgreSQL через TypeORM. Генератор нагрузки и протокольные клиенты также написаны на TypeScript. Таким образом, технологическая реализация соответствует профилю «Веб-технологии» и не переносит основную логику в отдельный Python-стенд.

## 3.2 Транзакционная обработка ставки

Команда ставки содержит `amount` и UUID `commandId`. Идентичность участника извлекается из проверенного Telegram initData; в исследовательском контуре разрешён заголовок demo-пользователя. На сервере выполняются следующие действия:

1. Начинается транзакция PostgreSQL.
2. Строка аукциона читается с `pessimistic_write`, что соответствует блокировке `SELECT FOR UPDATE` [7].
3. В таблице `processed_commands` выполняется поиск по составному ключу `(auctionId, commandId)`.
4. Проверяются статус, серверный дедлайн, запрет ставки владельца и ценовое правило.
5. Обновляются цена, лидер и `aggregateVersion`.
6. Сохраняются ставка, доменное событие и пользовательские уведомления.
7. В `processed_commands` сохраняется полный ответ команды.
8. После commit событие передаётся live-подписчикам, а уведомление — dispatcher бота.

Составной scope идемпотентности принципиален. Один и тот же UUID может легитимно встретиться в другом аукционе, поэтому глобальная уникальность `commandId` без `auctionId` способна вернуть ответ от чужого агрегата. Этот дефект был выявлен при разработке стенда и устранён составным первичным ключом.

PostgreSQL по умолчанию работает с изоляцией Read Committed [8]. Строчная блокировка сериализует изменения одного аукциона, не блокируя независимые лоты. Конкурирующая команда после ожидания получает уже обновлённую строку и повторно проходит ценовую проверку. Поэтому часть монотонно улучшавшихся команд в одной параллельной волне закономерно отклоняется: более сильное предложение может быть обработано раньше более слабого.

## 3.3 Модель данных и журнал событий

Основные таблицы — `auctions`, `bids`, `processed_commands`, `auction_events` и `notifications`. В `auction_events` действует уникальность `(auction_id, aggregate_version)`. Каждое событие содержит глобальный UUID, версию агрегата, тип, серверное время, correlation ID команды, JSONB payload и версию схемы.

Таблица 7 – Поля доменного события

| Поле | Назначение |
| --- | --- |
| `eventId` | Дедупликация доставки и сопоставление наблюдений |
| `auctionId` | Идентификатор агрегата |
| `aggregateVersion` | Строгий порядок изменений одного аукциона |
| `type` | `BID_ACCEPTED`, `AUCTION_EXTENDED`, `AUCTION_CLOSED` |
| `serverTimestamp` | Начальная точка измерения задержки |

Запись уведомления содержит `notificationId`, возрастающий `sequence`, адресата, `eventId`, `aggregateVersion`, текст, dedupe key, статус Telegram-доставки, число попыток и последнюю ошибку. Доменное событие и уведомления сохраняются внутри транзакции ставки или закрытия. Поэтому не возникает записи об уведомлении без причины и подтверждённого события без подготовленного inbox.

## 3.4 Протокол синхронизации

Endpoint `/auctions/{id}/sync` принимает `sinceVersion` и стратегию `snapshot`, `replay` либо `hybrid`. Для формирования согласованного ответа сервер сначала читает snapshot, затем журнал и отбрасывает события новее зафиксированной версии snapshot. Эта последовательность важна: параллельное чтение состояния и журнала способно вернуть разные точки времени.

Polling-клиент вызывает sync через заданный интервал. SSE endpoint сначала подписывается на live-поток, затем загружает историю и объединяет её с временным буфером. WebSocket gateway сначала присоединяет socket к комнате аукциона, выполняет recovery и после `sync:complete` освобождает отсортированный буфер.

Все клиенты реализуют дедупликацию по `eventId` и монотонной версии. Доставка допускает повтор на границе history/live, но повтор не создаёт эффект. Это соответствует практической схеме at-least-once поверх каналов с более слабой стандартной гарантией [3].

## 3.5 Контур уведомлений

После commit `NotificationService` публикует сохранённую запись в два независимых потребителя. WebSocket gateway направляет её в комнату адресата. Telegram dispatcher загружает запись из PostgreSQL и вызывает Bot API. Если адресат имеет числовой Telegram ID и отправка успешна, фиксируются `DELIVERED`, время и число попыток. При ошибке записывается `FAILED`; повтор планируется с экспоненциальной задержкой, а периодический worker дополнительно подбирает незавершённые записи после перезапуска процесса.

Глобальный inbox Mini App работает независимо от открытой карточки аукциона. Он выполняет первоначальную загрузку, подписывается на WebSocket, передаёт последний `sequence` после reconnect и периодически обновляет записи, чтобы показать статус Bot API. Кнопка в сообщении бота содержит deep link на соответствующий аукцион.

Внутри открытого аукциона уведомление проходит причинный барьер. Если `notification.aggregateVersion` больше локальной версии состояния, запись помещается в очередь. После snapshot или применения событий очередь проверяется повторно. Таким образом, сообщение о победе не отображается поверх экрана, который ещё считает торги активными.

Исследовательский адаптер Telegram сохраняет интерфейс `sendMessage`, добавляет 25 мс задержки и отклоняет первую попытку каждого сообщения. Retry начинается с 40 мс, максимальное число попыток равно пяти. Это не имитация всей сети Telegram, а контролируемая проверка состояния dispatcher и восстановления после временной ошибки. Реальный продуктовый контур использует тот же сервис с токеном BotFather.

## 3.6 Генератор нагрузки и сетевые нарушения

Для каждого trial создаётся новый аукцион длительностью 120 секунд с отключённым антиснайпером, чтобы версии зависели только от ставок и закрытия. Генератор строит монотонно улучшающийся массив предложений. Команды отправляются параллельными волнами. После основных команд часть уже принятых `commandId` отправляется повторно.

При reconnect-сценарии после заданной доли команд Toxiproxy отключает проксируемый канал. Сервер продолжает принимать оставшиеся ставки по контрольному адресу. Пока клиенты отключены, выполняются повторы команд и принудительное закрытие аукциона. Это гарантирует, что событие `AUCTION_CLOSED` и итоговые уведомления действительно пропущены live-клиентом. После выдержки proxy включается, а наблюдатели должны восстановить состояние.

Контрольный оракул формируется из последнего по качеству предложения и авторитетного журнала PostgreSQL. После сходимости стенд отдельно запрашивает snapshot, replay и hybrid от промежуточной версии. Уведомления всех участников проверяются на уникальность и связь с событием; для владельца работает полноценная модель Mini App с cursor и причинным барьером.

# 4 ПИЛОТНАЯ АПРОБАЦИЯ

## 4.1 Организация запуска

Финальный запуск имеет идентификатор `2026-07-22T17-51-14-522Z`. Выполнено 18 trial: три профиля, два вида аукциона и три повтора. Суммарная длительность измеряемых trial составила 72,928 секунды; один trial занимал от 0,894 до 7,240 секунды.

Создано 306 виртуальных клиентов: 102 polling, 102 SSE и 102 WebSocket. Генератор отправил 1008 исходных команд и 64 намеренных повтора. Из исходных команд 369 приняты, 639 отклонены после проверки актуальной цены. Получено 7041 сырое наблюдение доставки событий. Система создала 1125 уведомлений всем адресатам; модель Mini App владельца зафиксировала 387 доставок.

## 4.2 Дефект, выявленный пилотом

Первый полный запуск обнаружил важный дефект: один polling-клиент достиг финальной версии, но в его журнале отсутствовали четыре промежуточных события. Причина находилась не в polling, а в endpoint recovery. Snapshot и диапазон событий читались параллельно. При конкурентной записи запрос мог получить snapshot версии `v`, но журнал уже содержал событие `v + 1`. Проверка непрерывности отклоняла replay и возвращала snapshot, после чего клиент продолжал с версии `v`, не имея идентификаторов промежуточных событий в локальном журнале наблюдений.

Исправление установило согласованную верхнюю границу: сервер сначала читает snapshot, затем загружает события и оставляет только версии не выше `snapshot.version`. Добавлен регрессионный тест, моделирующий событие, появившееся после чтения snapshot. После очистки базы вся серия из 18 trial выполнена повторно. В финальном датасете итоговое число пропусков равно нулю.

Этот эпизод показывает, почему одного сравнения финальной версии недостаточно. Клиент мог выглядеть «сошедшимся», но не иметь полной истории. Комбинация `finalVersion`, множества `eventId` и авторитетного журнала позволила обнаружить ошибку измерительного и серверного контура до НИР-3.

## 4.3 Корректность конкурентных торгов

Во всех 18 trial фактическая цена и лидер совпали с ожидаемым лучшим предложением. Авторитетные версии были непрерывны. Все 64 повторные команды вернули сохранённый результат с признаком идемпотентного replay; число повторных доменных эффектов равно нулю.

Исходные команды обрабатывались в среднем 49,31 мс; медиана составляла 40 мс, p95 — 114 мс, максимум — 438 мс. Эти значения описывают текущую локальную среду и не используются как универсальный SLA. Важно, что рост конкуренции приводил к корректным отказам более слабых команд, а не к потере сильнейшего предложения.

Результат поддерживает H1 в исследованном диапазоне. Он одновременно подтверждает правильность прямого и обратного `BidPolicy`, транзакционной блокировки и составного scope идемпотентности.

## 4.4 Live-задержка транспортов

H2 оценивается только по `baseline-live`, поскольку при разрыве задержка включает политику reconnect и длительность fault. Для прямого аукциона медиана trial-p95 составила 266 мс для polling, 48 мс для SSE и 42 мс для WebSocket. Для обратного аукциона значения равны 267, 34 и 35 мс соответственно.

[[FIGURE:baseline-latency.png|Рисунок 4 – Медиана trial-p95 live-доставки в стабильном профиле]]

SSE и WebSocket в каждой baseline-группе оказались быстрее polling, что поддерживает H2. Разница близка к заданному интервалу polling и подтверждает корректность измерения окна устаревания. Между SSE и WebSocket преимущество мало и меняет направление между видами торгов; заявлять победителя среди этих двух транспортов по трём повторам нельзя.

## 4.5 Восстановление после разрыва

После reconnect все 306 клиентов завершили серию на серверной версии; итоговые `staleClients` и `missingEvents` равны нулю. Следовательно, H3 поддержана во всех 12 trial с разрывом.

В принудительном fault polling восстанавливался быстрее push-вариантов. Это не противоречит H2: live-доставка и reconnect — разные свойства. Polling возобновлял запросы по собственному короткому таймеру, тогда как EventSource и Socket.IO использовали политику переподключения. Полученный результат опровергает упрощённый тезис «WebSocket всегда быстрее» и показывает, что рекомендации должны учитывать не только live latency, но и параметры recovery.

## 4.6 Hybrid recovery

Во всех 18 контрольных проверках snapshot, replay и hybrid пришли к одной финальной версии, а replay содержал непрерывный диапазон. Hybrid во всех случаях выбрал snapshot. Размер snapshot находился в диапазоне 657–676 байт, replay — 2250–5675 байт. Таким образом, 18 из 18 решений совпали с минимумом JSON payload, и H4 поддержана как проверка корректности реализованной политики.

Однообразный выбор является одновременно ограничением пилота. Состояние аукциона мало, а контрольная точка бралась примерно на половине накопленной истории. Для исследования границы применимости необходимо варьировать размер snapshot, размер события и длину version gap. До такого расширения нельзя утверждать, что hybrid практически превосходит фиксированный snapshot.

## 4.7 Mini App и Telegram-уведомления

Система сохранила 1125 уведомлений. Исследовательский адаптер отклонил первую попытку каждого сообщения, поэтому все 1125 записей потребовали retry. В результате 1125 из 1125 получили статус `DELIVERED`; незавершённых `PENDING`, `FAILED` и `SKIPPED` не осталось. Trial-p95 времени доставки ботом находился в диапазоне 115–469 мс.

Модель Mini App проверяла уведомления владельца каждого аукциона. Получено 387 из 387 ожидаемых записей: 115 через live WebSocket и 272 через replay после cursor. Пропусков, повторных доставок в интерфейсе и нарушений причинного порядка не обнаружено. В baseline trial-p95 безопасного показа составлял 44–128 мс; в reconnect-сценариях, где Mini App был закрыт, — 1351–3507 мс и включал время fault и восстановления.

Эти сценарии должны войти в расширенную проверку.

## 4.8 Интерпретация результатов

Пилот подтверждает работоспособность слоистой архитектуры. Транзакционная блокировка и идемпотентность отвечают за результат торгов. Версионный журнал отвечает за восстановление клиента независимо от транспорта. Durable inbox отделяет пользовательское уведомление от жизненного цикла вкладки, а Telegram-бот создаёт внешний канал доставки.

Главный практический вывод заключается не в выборе одного транспорта. В стабильном соединении push-механизмы существенно уменьшили задержку относительно polling. После принудительного разрыва короткий polling восстановился быстрее настроенных push-клиентов. Следовательно, transport и recovery следует настраивать и оценивать отдельно. WebSocket подходит активному двунаправленному интерфейсу торгов, но его необходимо дополнять прикладным replay.

Telegram-бот также не заменяет синхронизацию состояния. Сообщение может сообщить о результате, но при открытии Mini App пользователь должен получить актуальный snapshot или события. Аналогично WebSocket не заменяет бота, потому что закрытый Mini App не поддерживает соединение. Наиболее устойчивым оказался комбинированный контур, в котором оба канала опираются на одну транзакционно сохранённую причину.

## 4.9 План НИР-3

НИР-3 должна расширить стенд и перейти от пилотной поддержки к статистически обоснованным рекомендациям веб-разработчикам. Планируются следующие шаги:

1. Увеличить число повторов на основании вариативности pilot dataset.
2. Рандомизировать порядок транспортов и сетевых профилей.
3. Варьировать размер snapshot, payload события и длину пропущенного диапазона.
4. Добавить ограничение пропускной способности и более длительные разрывы.
5. Запустить контрольную группу реальных Chromium Mini App клиентов.
6. Добавить CPU, RAM и event-loop lag как отдельные показатели.
7. Проверить перезапуск API и dispatcher, длительную ошибку Bot API и исчерпание retry.
8. Исследовать многопроцессный fan-out с Redis Streams либо другим сохраняемым адаптером.
9. Применить критерий Фридмана, post-hoc сравнения и оценки эффекта.
10. Сформулировать условия выбора polling, SSE, WebSocket и recovery, а не общий рейтинг технологий.

# ЗАКЛЮЧЕНИЕ

В НИР-2 спроектирован и реализован воспроизводимый экспериментальный стенд исследования конкурентных торгов и синхронизации состояния веб-клиентов. Прикладным сценарием стала Telegram Mini App платформа прямых и обратных аукционов рекламы, однако предмет исследования сохранён в области веб-архитектуры: транзакционная обработка команд, transport, recovery и доставка уведомлений.

Серверное ядро использует строчную блокировку PostgreSQL, повторную проверку ценового правила и составной идемпотентный ключ `(auctionId, commandId)`. Подтверждённые изменения получают монотонную версию и сохраняются в журнале. Polling, SSE и WebSocket используют единый envelope. Snapshot, replay и hybrid доступны через общий sync endpoint.

Для Telegram Mini App разработан двухканальный контур уведомлений. Запись создаётся в транзакции с доменным событием, активный клиент получает её через WebSocket, после reconnect восстанавливает по cursor, а Telegram dispatcher доставляет сообщение независимо от состояния Mini App и повторяет временно неуспешную попытку. Показ уведомления связан с `aggregateVersion`, поэтому пользователь не видит финальное сообщение поверх устаревшего состояния торгов.

В финальной пилотной серии выполнено 18 trial и подключено 306 виртуальных клиентов. Все торги завершились с правильной ценой и лидером. Все 64 повторные команды распознаны без второго эффекта. После разрывов не осталось устаревших клиентов и пропущенных событий. В стабильной сети SSE и WebSocket имели меньшую p95-задержку, чем polling. При reconnect polling в текущей конфигурации восстанавливался быстрее push-клиентов, что подтверждает необходимость раздельно оценивать live и recovery. Hybrid корректно выбирал меньший payload, но из-за малого состояния во всех опытах выбрал snapshot.

Все 1125 уведомлений были доставлены после искусственного отказа первой попытки Telegram. Mini App получил 387 из 387 ожидаемых уведомлений, включая 272 восстановленных после reconnect, без потерь, повторного показа и причинных нарушений. Таким образом, пилот поддержал пять сформулированных гипотез в тестируемом диапазоне.

Разработка стенда также выявила и позволила устранить дефект несогласованного чтения snapshot и журнала. Это подтверждает практическую ценность измерения не только финальной версии, но и полноты истории событий. Результатом НИР-2 являются программный стенд, методика, критерии и начальный датасет. Расширение выборки, временная cost model hybrid, браузерная контрольная группа и статистические рекомендации относятся к НИР-3.

# ПРИЛОЖЕНИЕ А. ВОСПРОИЗВЕДЕНИЕ ПИЛОТНОЙ СЕРИИ

Для запуска необходимы Node.js 22+, npm и Docker Desktop. Из корня репозитория выполняется команда `npm install`, затем `npm run research`. Команда поднимает отдельный compose-проект, ожидает готовность API и запускает 18 trial. Короткая smoke-проверка выполняется последовательностью `npm run research:up`, `npm run research:wait`, `npm run research:quick`.

Результат сохраняется в новом каталоге `datasets/<run-id>`. Путь последнего запуска находится в `datasets/latest.json`. Остановка контейнеров выполняется `npm run research:down`, полное удаление исследовательской базы — `npm run research:reset`.

Репозиторий проекта: https://github.com/iskandario/telebid-auction-lab.

# ПРИЛОЖЕНИЕ Б. СВЯЗЬ АРТЕФАКТОВ С ГИПОТЕЗАМИ

H1 проверяется по `commands.csv`, авторитетным событиям и итоговому snapshot. H2 использует строки `events.csv` и агрегаты baseline-профиля. H3 сопоставляет `clients.csv` с журналом и целевой версией после Toxiproxy reconnect. H4 сохраняет размеры и режим контрольных snapshot/replay/hybrid запросов в `summary.json`. H5 объединяет статусы всех сохраняемых уведомлений и строки `notifications.csv`, содержащие источник live/replay и момент безопасного показа.
